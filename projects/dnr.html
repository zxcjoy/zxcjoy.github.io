DNR Geospatial Risk Intelligence System

An end-to-end geospatial machine learning pipeline for predicting last-mile delivery failure (DNR — Delivered Not Received) risk across New York City. The system integrates Google Maps API geocoding, H3 spatial hotspot detection, config-driven feature engineering, and XGBoost + SHAP explainability — all orchestrated through a modular 9-command CLI with a SQLite caching layer.

Overview

What the system does and why it matters
  • Problem: NYC last-mile carriers face significant losses from DNR disputes — where customers claim packages were not received despite a delivery scan. Identifying high-risk addresses and zones before dispatch is critical.
  • Approach: Built a two-phase pipeline — Phase 1 handles API-heavy geocoding and enrichment with full caching; Phase 2 runs CPU-bound scoring, hotspot detection, and ML modeling without additional API spend.
  • Output: Each address receives a complete risk profile: 4 composite risk scores (0–100), an H3 spatial hotspot flag, a binary DNR probability from XGBoost, and SHAP explanations identifying the top contributing risk factors.
  • Design philosophy: Config-driven and modular — weights.yaml controls scoring formulas, poi_taxonomy.yaml defines POI categories, and each pipeline step can be toggled or rerun independently.

Pipeline Architecture

9-step, two-phase workflow from raw CSV to actionable risk intelligence
⬡ Phase 1 — Data Processing (API-Heavy, Cached)
STEP 01

Ingest & Clean

Standardizes raw CSV columns, removes invalid records, deduplicates tracking IDs, and constructs a full_address string. Outputs a clean deliveries table (~10K records).

PandasSQLAlchemy ORMClick CLI
STEP 02

Geocode

Extracts ~5K unique addresses and resolves each to lat/lng, place_id, and a location_type (ROOFTOP → APPROXIMATE) via the Google Geocoding API. A fingerprint-based SQLite cache (180-day TTL) eliminates redundant API calls on reruns — achieving 0 API calls after the first pass.

Google Geocoding APISQLite CacheMD5 Fingerprint
⬡ Phase 2 — Analysis (CPU-Bound, No API Calls)
STEP 03

Address Validation

Conditionally runs the Google Address Validation API on multi-unit addresses or DNR-flagged records. Checks for missing subpremise info and sets a structural_misdelivery_risk baseline.

Google Address Validation APIConditional Logic
STEP 04

Roads Snap

For addresses below ROOFTOP precision, snaps coordinates to the nearest road using the Google Roads API. Computes offset_to_road_m — a proxy for access complexity — while skipping the call for already-precise addresses to save API cost.

Google Roads APICost-Aware Routing
STEP 05

Places POI Enrichment

Queries Google Places API at 200m and 500m radii around each address to count nearby points of interest. Categories are defined in poi_taxonomy.yaml: educational, medical, commercial, and high-mobility (transit, cafes, parks) — inputs to the porch piracy vulnerability score.

Google Places APIpoi_taxonomy.yaml200m / 500m Radius
STEP 06

Risk Scoring

Computes four composite risk scores (each normalized to 0–100) using weighted formulas defined in weights.yaml. Scores are fully configurable without code changes.

weights.yamlMin-Max Normalization
STEP 07

Hotspot Detection

Aggregates delivery records into H3 grid cells (resolution 9 ≈ ~0.1 km²). Applies Laplace smoothing to DNR rates, then Z-score thresholding (z ≥ 1.96, 95% confidence) to flag blacklist zones. Outputs 37 statistically significant micro-hotspot cells as GeoJSON for routing integration.

STEP 08

Predictive Modeling

Trains an XGBoost binary classifier (with scale_pos_weight for class imbalance) on the full feature store. SHAP TreeExplainer generates per-prediction force plots and global feature importance rankings. Metrics include AUC, PR-AUC, and calibration checks.

STEP 09

Reporting

Exports the full address feature store as Parquet, hotspot GeoJSON, model metrics JSON, SHAP summary CSV, and interactive Folium HTML maps — ready for stakeholder delivery or routing system ingestion.

Composite Risk Scores

Four address-level risk dimensions, each normalized to a 0–100 scale
🔵 Geo Reliability
Reflects geocoding precision. ROOFTOP scores highest; approximate or interpolated results indicate structural ambiguity in the address.
Input: location_type from Geocoding API
🟠 Structural Misdelivery Risk
Captures address structural issues — missing apartment/unit info, incomplete subpremise data, or addresses flagged by the Validation API.
Input: Validation API + geocoding metadata
🟡 Access Complexity Index
Measures physical access difficulty using road offset distance. Addresses far from snapped road coordinates suggest gated, private, or hard-to-reach locations.
Input: Roads API offset_to_road_m
🔴 Porch Piracy Vulnerability
Estimates theft exposure based on high-mobility POI density — transit hubs, commercial centers, and foot-traffic generators within 200–500m of the delivery address.
Input: Places API POI counts

Key Achievements

Quantified outcomes from the pipeline
  • 100% geocoding coverage across 5K unique NYC addresses — 99.6% at ROOFTOP precision (Google's highest-accuracy tier).
  • Zero redundant API calls on pipeline reruns, achieved via an MD5 fingerprint-based SQLite cache with a 180-day TTL.
  • 50+ features per address stored in a Parquet feature store, combining geocoding metadata, POI density, road proximity, and spatial aggregation signals.
  • 37 H3 hotspot cells identified at 95% statistical confidence (z ≥ 1.96), covering zones with DNR rates up to 3.6× the citywide baseline.
  • Per-prediction SHAP explanations surfaced in interactive HTML reports (165–242 KB), revealing the top geographic risk drivers for non-technical stakeholders.
  • Modular two-phase CLI with 9 commands — clean separation of API-heavy Phase 1 and CPU-bound Phase 2 enables incremental reruns without redundant API spend.

Smart API Caching
